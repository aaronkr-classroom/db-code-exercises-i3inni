## 연습문제 정답

### 1. 개체 통합이 가능하지 않은 것은?

**정답: 모든 속성의 도메인이 동일한 개체 타입들**

> 개체 통합은 기본키를 기준으로 이루어진다. 기본키가 동일하거나, 상호 식별자가 되거나, 기본키와 속성이 비슷한 경우에 통합이 가능하다. 단순히 모든 속성의 도메인이 같다고 해서 통합 대상이 되는 것은 아니다.

### 2. 이력 데이터에 대한 설명이 잘못된 것은?

**정답: 변경 이력은 데이터가 변경될 때마다 변경 후의 상태를 확인해야 할 때 발생한다.**

> 변경 이력은 변경 **전후의 차이**를 확인하기 위해 남기는 것이다. "변경 후의 상태만" 본다는 표현은 잘못됐다.

### 3. 개체 무결성 제약조건이 만족해야 하는 세 가지 규칙에 해당하지 않는 것은?

**정답: 기본키는 데이터 보안이 요구되어야 한다.**

> 개체 무결성의 세 가지 규칙:
> - 기본키 구성 속성은 NULL을 가질 수 없다
> - 기본키는 유일성을 보장하는 최소한의 속성 집합이어야 한다
> - 기본키는 개체를 유일하게 식별할 수 있어야 한다
>
> 데이터 보안은 무결성과는 별개의 개념이다.

### 4. 참조 무결성에 대한 설명이 잘못된 것은?

**정답: 외래키는 널 값을 가질 수 없다.**

> 외래키는 NULL 값을 가질 수 **있다**. 단, 값을 갖는다면 반드시 피참조 릴레이션의 기본키 값 중 하나여야 한다.

### 5. 참조 무결성의 세부 업무 규칙에 대한 설명이 잘못된 것은?

**정답: 제한(Restrict): 대응하는 자식 인스턴스가 없는 경우 부모 인스턴스의 삭제/수정이 불가능하다.**

> 제한(Restrict)은 대응하는 자식 인스턴스가 **있는** 경우에 부모 인스턴스의 삭제/수정이 불가능한 규칙이다. "없는 경우"가 아니라 "있는 경우"가 맞다.

### 6. 명시적으로 정의해야 하는 도메인 무결성에 해당하지 않는 것은?

**정답: UNIQUE 제약조건**

> 도메인 무결성은 속성 값의 범위·형식·기본값 등에 관한 제약이다. NULL, CHECK, DEFAULT는 도메인 무결성에 해당하지만, UNIQUE는 **키 무결성**에 속한다.

### 7. 데이터 표준화의 일반 원칙이 잘못 설명된 것은?

**정답: 서술적 정의만으로 의미 전달이 가능하도록 상세히 작성한다.**

> 데이터 표준화는 서술적 정의에만 의존하면 안 되고, 표준 단어·도메인·코드 등을 함께 활용해 일관성을 확보해야 한다.

### 8. 데이터 표준의 주요 대상에 해당하지 않는 것은?

**정답: 표준 개체**

> 데이터 표준의 주요 대상은 표준 단어, 표준 용어, 표준 도메인, 표준 코드이다. "표준 개체"라는 항목은 없다.

### 9. 데이터 표준화 절차가 바르게 작성된 것은?

**정답: 요구사항 수집 → 데이터 표준 정의 → 데이터 표준 확정 → 데이터 표준 관리**

> 일반적인 표준화 절차는 수집 → 정의 → 확정 → 관리 순서로 진행된다.

### 10. 고객구분코드를 릴레이션 스키마로 변환

**고객구분코드 릴레이션**

```
고객구분코드(고객코드, 실적기준, 할인율, 제공마일리지)
```

| 고객코드 (PK) | 실적기준 | 할인율 | 제공마일리지 |
|---|---|---|---|
| Green | 1천 이상 | 5% | 300만 원 |
| Orange | 2천5백 이상 | 5% | 500만 원 |
| Silver | 5천 이상 | 7% | 700만 원 |
| Gold | 7천 이상 | 10% | 1,000만 원 |
| Black | 1억 이상 | 10% | 2,000만 원 |

- **기본키**: 고객코드

### 11. 클럽 회원의 행사 참석 내역 ER → 릴레이션 스키마

> ※ 교재 351쪽 ER 다이어그램을 보지 못해 일반적인 형태(회원 ─ 참석 ─ 행사, 다대다 관계)로 가정해 변환합니다.

```
회원(회원번호, 이름, 연락처, ...)
행사(행사번호, 행사명, 행사일자, 장소, ...)
참석(회원번호, 행사번호, 참석일시, ...)
```

- **회원** PK: 회원번호
- **행사** PK: 행사번호
- **참석** PK: (회원번호, 행사번호) — 복합키
  - FK: 회원번호 → 회원, 행사번호 → 행사

### 12. 아파트 재건축 공사 진행 상황 관리

**ER 다이어그램 구조**

- **개체**: 아파트, 진행단계
- **관계**: 아파트 ─ (1:N) ─ 진행단계 (하나의 아파트는 여러 진행단계를 가짐)
- 진행단계는 아파트에 종속된 **약한 개체**로 볼 수 있음 (단계 구분: 계획·준비·철거·건축·마무리·입주)

```
[아파트] ──< 진행 >── [진행단계]
   1                       N
```

**릴레이션 스키마**

```
아파트(건물번호, 이름, 주소, 시공사, 세대수, 입주시작일)

진행단계(건물번호, 순서, 주관사, 연락처, 시작일자, 종료일자)
```

- **아파트** PK: 건물번호
- **진행단계** PK: (건물번호, 순서) — 복합키
  - FK: 건물번호 → 아파트(건물번호)

### 13. 주식 주가 변동 이력 관리

**ER 다이어그램 구조**

- **개체**: 주식, 주가이력
- **관계**: 주식 ─ (1:N) ─ 주가이력 (하나의 주식은 여러 변동 이력을 가짐)
- 주가이력은 **변경 이력**에 해당 (변동 발생 시점만 관리)

```
[주식] ──< 변동 >── [주가이력]
   1                    N
```

**릴레이션 스키마**

```
주식(종목코드, 종목명, 업종, 총주식수)

주가이력(종목코드, 변동일시, 주가)
```

- **주식** PK: 종목코드
- **주가이력** PK: (종목코드, 변동일시) — 복합키
  - FK: 종목코드 → 주식(종목코드)
